Design Documentation

Google Ads API Access Application · Med4Grow

Note: This document describes the architecture, intended usage and mockups of the Med4Grow platform's integration with the Google Ads API.

Company Name: Med4Grow (commercial brands: MedDesk for medical verticals, WakeDesk for general business verticals).

Business Model

Med4Grow is a B2B SaaS platform that provides medical clinics, dental clinics and small/medium-sized businesses with a complete operational system: CRM and sales funnel for inbound leads, smart scheduling, electronic medical records, WhatsApp Business integration with an AI assistant, financial control, and consolidated marketing analytics.

We do not operate as a media agency and we do not run paid ads on behalf of third parties. Each of our customers owns their own Google Ads account. Through an opt-in flow, the customer grants Med4Grow read access to their account (via MCC link) so that performance metrics can be displayed within the Med4Grow platform alongside Meta Ads spend and organic WhatsApp leads. The Google Ads API is used strictly for reporting and read-only synchronization; we do not create, edit or manage campaigns on behalf of our clients through the API.

We do use Google Ads ourselves as an advertiser, to acquire new clinics as Med4Grow customers (Search, Performance Max and YouTube campaigns). That advertising activity uses our own account directly through the Google Ads UI, not through the API.

Tool Access / Use

The Med4Grow web application is the only place from which the Google Ads API is called. It is used by three audiences:

  • Clinic owners and managers (our paying customers): view dashboards and download CSV / PDF reports of their own ads performance, side-by-side with their leads pipeline and revenue.
  • Org-admin users from organizations that group multiple clinics: view aggregated performance across the clinics they administer.
  • Internal Med4Grow support staff (super-admin role): troubleshoot and provide assisted onboarding.

Each customer authorizes access to their own Google Ads account explicitly. The platform stores the linked account ID and an MCC token in our database (Supabase, hosted on AWS). Customers can revoke access at any time from their settings page. No third party outside the customer's own organization can view their data.

A scheduled background job runs every hour, pulling updated performance metrics from the Google Ads API into our database so dashboards stay fresh without manual refresh. The same job pauses synchronization if the customer revokes access or downgrades their plan.

Tool Design

Architecture:

  • Frontend: React + TypeScript single-page application served from our domain. Renders dashboards and reports.
  • Backend: Supabase (PostgreSQL + edge functions, written in TypeScript/Deno).
  • Sync layer: An hourly edge function (a CRON-triggered worker) reads the list of opted-in clinic accounts, calls the Google Ads API on behalf of each one, normalizes the response and writes the metrics into the marketing_data table.
  • UI layer: The frontend queries marketing_data through Supabase's REST/Realtime layer to draw charts and tables. Users can filter by date range, by campaign or by clinic. PDF and CSV exports are generated client-side from the same data.

All access is gated by Supabase Row-Level Security policies, so a clinic can only ever see its own performance data. Tokens used to talk to the Google Ads API are stored encrypted at rest.

API Services Called

  • GoogleAdsService.SearchStream — to retrieve performance metrics (impressions, clicks, conversions, cost) at account, campaign and ad-group granularity via GAQL queries.
  • CustomerService — to validate the linked customer ID and surface human-readable account names in our UI.
  • CustomerClientService — to enumerate the child accounts available under our MCC for the onboarding flow.
  • CampaignService (read-only) — to list campaigns and their status for the campaign-level performance views.

No mutating endpoints are called. We do not create, edit, pause or delete entities on behalf of our customers through the API.

Background Job Description

Our SQL/Supabase backend syncs with the Google Ads API hourly to update each opt-in customer's performance metrics. The job:

  • Reads the list of clinics with google_status = 'active' in our database;
  • For each, calls GoogleAdsService.SearchStream with a GAQL query scoped to the last 90 days of metrics;
  • Upserts the rows into marketing_data (clinic_id, date, platform, investment, conversions_value);
  • Logs the run, surfaces errors to the support dashboard, and retries with exponential backoff on transient failures.
